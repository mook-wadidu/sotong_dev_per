## ★ MVP 범위 (이번 출시) — Now / Next / Later

> 이 PRD는 **전체 설계도**다. 이번에 짓는 건 **1층(MVP)**뿐. 아래 NOW만 만들면 출시. 나머지는 전부 검증 후.

### NOW (MVP — 이것만, 5개)

| # | 기능 | Acceptance Criteria (Done 기준) |
|---|---|---|
| 1 | **QR 스캔 → 다국어 입력** (살롱 고정 QR 1개) | QR 스캔 후 **3초 내** 로드 · **1차 영어·일본어**(중국어는 NEXT) · 무설치·무가입 |
| 2 | **AI 한국어 요약** | 입력 제출 후 **15초 내** 요약 생성 · 디자이너가 **수정 없이 사용 가능한 비율 ≥ 80%**(파일럿 측정) |
| 3 | **디자이너 확인** | 요약을 **공용 태블릿/디자이너 기기**에 표시(푸시 없이 새로고침/세션목록 조회) · 보고 응대 가능 |
| 4 | **결과 입력 (사진 중심)** | 비포/애프터 **사진 2장 필수** · 시술 탭 선택 · 텍스트는 옵션 · 입력 **30초 이내**(파일럿서 실측·검증) |
| 5 | **재방문 자동 감지 + 지난 기록 표시** | 같은 손님 재방문 시 지난 기록 자동 로드 · 지난 시술이 기본값으로 표시(같으면 탭 1번) |
| 0 | **손님 세션 앵커 (기반)** | QR 세션 + **기기 토큰**으로 "이 손님 = 이 세션" 고정. 동의·사진·기록·재방문이 한 세션에 묶임 |

→ 이 5개 + 기반(#0)이 "상담 통함 + 데이터 쌓임"의 최소 루프. 끝.

**⚠️ MVP 신원 모델 (A·B를 한 번에 푸는 핵심)**: NOW #5 재방문 인식과 NOW #4 사진-동의 결합은 **같은 문제** — "이 손님 = 이 세션"을 묶을 앵커가 필요하다. 그래서 **기기 토큰만 NEXT→NOW로 끌어온다**(OTP·전화번호 인증은 NEXT 유지). 손님이 자기 폰에서 QR로 인테이크+동의 → 그 폰의 기기 토큰이 세션 앵커 → 디자이너가 찍는 사진·기록·재방문이 모두 이 토큰에 묶인다. 번호 인증 없이도 *같은 기기 재방문*은 인식 가능(§10.1의 "미인증 번호 이력 로드 금지"와 충돌 안 함 — 번호가 아니라 기기 토큰 기반).

### ★ MVP Success Metric (첫 달 목표 — 가설, 파일럿이 검증)
```
살롱        : 3곳   ← 패턴 반복 확인에 필요한 최소 표본(1곳=우연, 3곳=경향)
외국인 상담  : 100건  ← 살롱당 ~33건, 데이터 패턴이 보이기 시작하는 최소량
완결 기록    : 80%   ← 데이터 품질 최소 기준. 이 밑이면 §12.2 입력 동기 실패 신호
```
→ 이 셋이 첫 달 합격선. *목표 가설이며 파일럿 실측으로 보정(§15.1).* **80% 완결률이 가장 중요** — 상담만 되고 결과 입력이 안 되면 모델 연료가 안 쌓인다. 완결률이 곧 "디자이너가 실제로 입력하는가"의 측정치.

### ★ 북극성 지표 (North Star KPI) — "상담 성공률"
```
상담 성공률 = (상담·시술까지 이어진 외국인) / (방문한 외국인)
   기존(왓츠앱·파파고 땜빵) : TBD  ← 파일럿 측정
   소통 적용 후            : TBD  ← 파일럿 측정 (목표: +30%p 이상 향상)
```
→ **이 한 줄이 모든 기능을 설명한다.** 말 안 통해 돌려보내던 외국인(실패)을 상담 성공으로 바꾸는 것 = 사업의 본질. 분산된 지표(§15) 대신 이걸 북극성으로. *숫자는 지어내지 않는다 — 기존값·향상폭 모두 파일럿 실측으로 채운다.*
> **분모 측정법**: "방문했지만 그냥 돌아간 외국인"은 시스템에 안 잡힌다 → 파일럿 기간 **디자이너 수기 집계(도어카운트)**로 분모를 센다(들어왔으나 상담 못 한 외국인 수를 디자이너가 태그). 자동 집계는 NEXT.

### ⚠️ 주연/조연 규율
**돈 버는 주연은 'QR 다국어 상담'이다. AI 모델(판단·생성·B2B)은 조연이자 미래다.** AI 이야기가 재밌어도 주객전도 금지 — 지금 매출과 검증은 QR 상담에서 나온다. 화려한 AI에 끌려 첫 파일럿을 미루지 않는다.

### LATER (데이터 쌓인 뒤 — V2+)

**규율**: NOW에 없는 기능은 MVP에서 *만들지 않는다*. "있으면 좋은 것"에 끌려 출시를 미루지 않는다 — 부끄러울 때 내보낸다.

## 0. 한 줄 정의 & 피치

**슬로건**: **K-뷰티 시술 개인화 모델.**

**한 줄**: *손님은 자기 언어로 탭하고, 디자이너는 한국어 요약을 받는다.*

소통은 한국 살롱을 위한 **다국어 AI 상담·접수 데스크 + 시술 기록(카르테) 엔진**이다. 외국인은 QR로 자기 언어 접수 → AI 한국어 요약 → 번역 상담 → 손님 언어 리포트. 한국인 단골은 같은 흐름으로 시술 이력이 쌓여 디자이너가 기억할 필요가 줄어든다. 설치 5분. (가격 미확정)

**본질**: (1) 살롱-외국인 사이 통역 접수원 + (2) 전 고객 시술 결과를 쌓는 데이터 엔진.

---

## 1. 배경 & 문제

방한 외국인 역대 최대, 소비가 일상(미용실·올리브영·피부과)으로 이동. 살롱은 **문의·접수에서 외국인을 놓침**(말 안 통해 돌려보냄/오상담). 동시에 단골 관리에서도 "지난 시술·약제 기억" 부담이 상존. 기존 대안(영어 직원·통역·범용 번역 앱, 종이 카르테)은 비싸거나 불완전.

## 2. 비전 & 포지셔닝

- **북극성 / 슬로건**: **K-뷰티 시술 개인화 모델.** (외국인이 한국에서 받는 뷰티 서비스의 다국어 접수·상담 인프라에서 출발.)
- **회사 정체**: 미용실 관리 SW 회사가 아니라 **외국인 K-뷰티 경험의 데이터·AI 인프라 회사**.
- **웨지 규율 (중요)**: 진입·피칭·차별화는 **외국인 다국어 상담**(빈 슬롯·블루오션)으로. 한국인/전 고객 카르테는 *간판이 아니라* 들어간 뒤 자라는 확장(끈끈함·데이터·객단가↑). 간판을 '전 고객 카르테'로 갈면 헤어짱(17,000)·LiME류·네이버와 정면충돌하는 레드오션 + '관리 도구' 선반 추락.

## 4. 우리가 만드는 AI

> **한 줄 (VC용)**: 지금은 기존 LLM으로 다국어 상담을 푼다. 진짜 자산은 그 과정에 쌓이는 *실제 한국 살롱 시술 결과 데이터*이고, 그게 쌓이면 "이 시술 → 이 결과"를 판단하고 시각화하는 모델이 된다 — 범용 AI가 못 가진 데이터라 베낄 수 없다.
>
> *(아래는 구현 디테일 — 개발용. VC/PM은 위 한 줄로 충분.)*

세 층으로 진화한다.

**(1) 지금 — 상담 보조 AI (얕은 층, Gemini).** 인테이크 → 디자이너용 한국어 요약, 미용 용어 맥락의 양방향 번역, 헤어 리포트 초안. *눈에 보이는 제품이지만 해자는 아님.*

**(2) 짓고 있는 것 — 결과 근거 추천 엔진.** 손님의 실제 얼굴·모발 + 원하는 방향을 입력으로, **한국 살롱이 실제로 시술한 결과 데이터에 근거해** "이 시술을 하면 이렇게 되고 만족도는 이 정도"를 추천한다. "어울려 보인다"(의견)가 아니라 "실제로 해봤더니 이랬다"(결과). retrieval-first — 레코드 1개부터 가치, 임계치 없음. "동양인 최적 모델" 같은 일반론 아님.

**(3) 최종 — K-뷰티 시술 결과 예측 모델.** 얼굴·모발만 보고 시술/스타일과 *그 결과·만족도*를 예측·추천하는, 세상에서 가장 큰 '실제 한국 살롱 시술 결과' 데이터로 학습된 모델. 이게 곧 **K-뷰티 개인화 레이어** — 업계는 일반론으로 가짜 개인화를 하지만, 우리만 *실제 결과*로 진짜 개인화를 한다. 적용 범위·엔드 비전은 §17.

**해자**: 얼굴·모발 → 실제 시술·약제 → 비포·애프터 → 결과가 구조화 누적. 범용 AI가 못 가진 데이터이고, 쌓일수록·시간이 걸릴수록 방어된다. 한국인 단골까지 받으면 축적 속도↑.

**지금 우선순위**: 화려한 모델이 아니라 **캡처 품질**(일관 분류·비포애프터·결과점수) — 이게 (2)(3)의 연료.

### 4.1 기술 구현 노트 — LLM ≠ 최종 모델
- **1층 (지금)**: 기존 LLM API(Gemini 등) + 프롬프트 엔지니어링. 번역·요약·리포트 초안. *밑바닥부터 모델 안 짬.*
- **2층 (1~2년)**: 벡터 검색 + 자체 DB **retrieval**. LLM 아님. 레코드 1개부터 가치.
- **3층 (최종, 판단)**: 비전 + 구조화 데이터 **예측 모델**. LLM 아님. 파운데이션 모델을 빌려 소통 데이터로 **파인튜닝**. *한국 특화인 이유 = 학습 데이터가 전부 한국 살롱 결과라서이지, 특별한 아키텍처 때문이 아님.* "이 얼굴·모발 + 이 시술 → 실제로 이런 결과·만족도"를 **판단**한다.
- **4층 (시각화, 생성)**: 3층 판단 모델이 정한 결과를 **기존 손님 얼굴에 편집·생성**해 보여주는 레이어. 생성 모델(diffusion 등)은 *제멋대로 예쁘게 합성하지 않는다* — 판단 모델이 "실제 데이터상 이렇게 나온다"고 정한 결과에 **종속**되어 그것을 그려낼 뿐(판단 모델의 통역사).

**판단 → 생성 파이프라인 (핵심 설계)**:
```
판단 모델(3층) — "실제 데이터상 이 시술 → 이 결과"  ← 무엇을
        ↓
생성 모델(4층) — 그 결과를 기존 얼굴에 편집·시각화  ← 그것을
```
- **의존 순서 (규율)**: 생성(4층)은 판단(3층)이 먼저 돌아간 다음에야 의미가 있다. 판단 없는 생성 = 일반 헤어 합성 앱(실제와 무관). **반드시 판단 3층 → 그 위에 생성 4층** 순서.
- **해자**: 일반 합성 앱은 생성 모델이 혼자 "예쁘게" 합성 → 실제와 무관. 우리는 **실제 한국 살롱 결과 데이터가 생성의 고삐를 쥔다** → 기대-실제 갭(클레임 리스크)이 작고, 데이터 없이는 복제 불가.

### 4.2 캡처 품질 4원칙 (더티 데이터 방어)
1. **구조화 강제** — 자유 입력 최소화, 분류·태그로 받기.
2. **앵커 척도** — 모발 상태·손상도 등 일관된 기준 척도.
3. **사진 = 객관 정답** — 비포/애프터 사진이 결과의 그라운드 트루스.
4. **디자이너 디바이어싱** — 디자이너 주관 평가를 보정하는 feature 설계.

## 5. 시장 (TAM / SAM / SOM)

### 5.1 도메인 확장 경로 (TAM 확대)
헤어는 *문*이고, 같은 모델·데이터 구조가 뷰티 전반으로 확장된다.
- **Phase A (지금)** — 헤어
- **Phase B** — 네일 · 속눈썹 · 왁싱 · 메이크업 (같은 패턴, 도메인마다 새 데이터)
- **Phase C** — 피부과 · 성형외과 (의료미용) — 시장 가장 크나 *별도 데이터 + 규제*(외국인환자 유치업 등)

→ 최종 TAM = **K-뷰티 시술 전반** (외국인 + 한국인). 단 **각 도메인은 공짜 전이가 아님** — 데이터는 도메인마다 새로 쌓아야 하고 의료는 규제. 순서는 *헤어 깊이 먼저*, 확장은 데이터 증명 뒤.

### 5.2 경쟁 구도 — "레이어가 다르다"

핵심 주장: **외국인 미용 수요는 폭발 중인데, "살롱 내 다국어 상담·접수"를 푸는 직접 경쟁자가 비어 있다.** 기존 플레이어는 전부 다른 레이어에 있다.

| 플레이어 | 레이어 | 하는 것 | 못 하는 것 (소통 차별점) |
|---|---|---|---|
| **크리에이트립** | 발견·예약 (방문 前) | 월 150만 외국인 대상 미용실·피부과 발견·예약·콘텐츠 | 살롱 *안*에서의 실시간 상담·시술 데이터 X. 광고/입점 모델 |
| **클룩·마이리얼트립·트립닷컴 등 OTA** | 상품 예약 (방문 前) | K-뷰티 체험 상품 패키지 판매 (7.7만~30만) | 개별 살롱 상담·카르테 X. 일회성 관광상품 |
| **네이버·카카오 예약** | 국내 예약 | 한국인 예약·리뷰 | 외국인 미지원 (그래서 살롱이 왓츠앱·인스타 DM으로 땜빵) |
| **왓츠앱·인스타 DM** | 상담 (현행 땜빵) | 외국인 살롱이 실제로 쓰는 수작업 채널 | 수작업·비구조화·데이터 0. **소통이 대체하려는 바로 그 지점** |
| **파파고·ChatGPT** | 번역 (현장) | 미용사가 즉석 번역에 사용 | 미용 도메인 특화 X, 시술 데이터 축적 X, 구조화 X |
| **Zenoti·Mindbody** | 글로벌 살롱 SaaS | 예약·POS·CRM (글로벌 대형) | 외국인 다국어 상담 특화 X, 한국 시술 데이터 X |
| **헤어짱·LiME** | 국내 살롱 관리 | 한국 살롱 예약·고객관리 | 외국인 상담 X (한국인 대상) |

**구조적 빈틈**: 발견·예약(크리에이트립)과 관리 SaaS(헤어짱) 사이, **"살롱에 외국인이 들어온 그 순간의 상담·접수·시술기록"** 레이어가 비어 있다. 현장은 이걸 왓츠앱 DM + 파파고로 수작업 땜빵 중(한국경제·2026.3 보도: 홍대 T미용실 등). 소통은 이 빈 칸을 직접 채운다.

**복제 위험 / 해자**: 도구 자체(다국어 상담 폼)는 베끼기 쉽다. 방어선은 *도구가 아니라* — ① 시술 결과 데이터 축적(§4) ② 살롱별 락인 ③ 시장 진입 속도 ④ 판단→생성 모델(§4.1). 크리에이트립이 150만 유저로 위에서 내려와도, *살롱 내 시술 결과 데이터*는 현장 도구를 깐 자만 가진다.

> 캐비엇: "직접 경쟁자 없음 = 블루오션"은 *매력적이지만 위험한 가정*. 수요가 크면 누군가 들어온다. 방어는 "경쟁자가 없다"가 아니라 "데이터를 먼저·빨리 쌓는다"로 한다.

### 5.3 경쟁사가 대응하면? (반격 시나리오)

| 시나리오 | 우리 답 |
|---|---|
| **크리에이트립**이 QR 상담 기능 추가 | 걔넨 발견·예약(방문 前)이 본업 → 살롱 *내부* 워크플로우·시술 데이터는 현장 도구를 깐 자만 가짐. 우리가 이미 깐 살롱은 락인. 오히려 *예약은 걔네 → 상담·기록은 우리* 협업 가능 |
| **헤어짱·LiME**가 번역 붙임 | 번역은 commodity(파파고 수준). 해자는 번역이 아니라 *시술 결과 데이터 + 판단·생성 모델*. 한국인 관리툴이 외국인 워크인·결과 데이터를 따라잡으려면 시간이 걸림 = 우리 선점 구간 |
| **ChatGPT**가 살롱 전용 GPT 출시 | 범용 LLM은 *약제 비율·모발 인장강도·실제 시술 결과*를 모름(§18 VC 한 문장). 우리 데이터 없이는 "결과 근거 추천" 불가 |

**공통 방어선**: 도구는 다 베껴진다 → 진짜 해자는 ① 누적된 *실제 시술 결과* 데이터 ② 살롱별 락인 ③ 진입 속도. 경쟁 대응의 정답은 방어적 기능 빌드가 아니라 **더 빠른 현장 실행**.

### 5.4 정량 가설 (파일럿이 채울 빈칸 — §15.1과 연동)

VC가 가장 먼저 묻는 건 "Retrieval 어떻게?"가 아니라 **"외국인 하루 몇 명? 몇 명 놓침? 돈 얼마?"**. 현재는 가설(숫자는 파일럿 실측으로 교체):

```
홍대 외국인 밀집 살롱 (예시 가설, 미검증)
  └ 외국인 일 평균        : N명        ← H1 실측 필요
      └ 말 안 통해 상담 실패 : M명        ← H2 실측 필요
          └ 외국인 객단가     : 10만+      ← 시장 평균(명동 커트 2.5~4.5만·헤드스파 8~15만)
              └ 월 기회손실   : M × 객단가 × 영업일
```
→ 이 손실의 일부만 막아도 구독료 회수. **단, N·M은 반드시 파일럿 실측으로 채운다**(지금은 빈칸). 지어낸 숫자를 단정으로 쓰지 않는다.

## 6. 타깃 사용자 & 페르소나 *(갱신)*

**계정/신원을 갖고 '쓰는' 주체 = 살롱(디자이너·사장) + 소통 운영(나). 손님은 로그인 없는 사용자.**
손님 = **외국인 워크인 + 한국인 단골**(전 고객).

| 페르소나 | 신원 | 화면 | 하는 일 |
|---|---|---|---|
| **손님 (외국인/한국인)** | 없음. 기기 토큰 또는 **OTP 인증된 전화번호** | 모바일 웹 (QR) | 언어 선택 → 탭 접수 → (외국인)번역 상담 → 리포트 수신 |
| **디자이너** | 일회성 기기 등록 (가벼운 신원, salon 묶임) | PWA + 자체 푸시 | 알림 받기 → 요약 → 칩 응답 → 30초 기록 → 리포트 발송 |
| **사장 (owner)** | 계정 (네가 온보딩) | 웹/어드민 | 직원·메뉴·가격·QR 관리, 다축 조회 |
| **소통 운영 (platform admin)** | 계정 | 어드민 대시보드 | 살롱 온보딩, 전 살롱 모니터링, 데이터·지표 |

손님은 계정 유저는 아니나 **손님 경험이 곧 제품**이며, 리포트·전화번호 식별·(나중)회원이 "손님 = 우리 유저" 전환 씨앗.

## 7. 핵심 가치 제안

- **손님**: 내 언어로 막힘없는 상담 + 내 언어 리포트. 단골은 "지난번처럼" 한 번에.
- **살롱**: 외국인 = 신규 매출(웨지) + 전 고객 카르테로 디자이너 기억 부담↓(락인). 외국인 1~2명이면 회수.
- **영업 한 줄(진입)**: "외국인 손님 말 안 통해 돌려보낸 적 있죠? QR 하나면 손님이 자기 언어로 접수하고, 디자이너님은 한국어 요약만 받습니다."

## 8. 사용자 여정 & QR 플로우 *(갱신)*

**QR 메커니즘**: 두 종류를 병행한다 — 워크인 마찰 없이, 단골은 정확하게.

- **살롱 고정 QR (기본)** — `sotong.ai/c/{salon_token}`. 입구·대기·테이블텐트 비치. **워크인 외국인용**(어느 디자이너인지 모름 → 고민 없이 아무거나 찍으면 됨). 찍으면 브로드캐스트 → 먼저 받는 디자이너 연결(or 사장 배정).
- **디자이너 개별 QR (옵션, admin 발급)** — `sotong.ai/c/{salon_token}/{designer_token}`. 디자이너가 명함·인스타·카톡에 공유. **단골·지명 손님용** → 찍으면 그 디자이너로 라우팅 고정. 디자이너가 자기 손님 끌어오는 마케팅 도구로도 활용.

*규율: 살롱 QR을 **대체하지 않고 추가**. 워크인 진입 단순함(살롱 QR)을 절대 깨지 않음. designer_token 없으면 기존 브로드캐스트 동작.*
*MVP 범위: 데이터 구조에 `designer_token` 자리만 마련. 실제 발급 UI는 디자이너 多 살롱 진입 시점에 활성화(파일럿 1~2명 살롱은 살롱 QR로 충분).*

**첫 방문**
1. QR 스캔 → 모바일 웹(무설치·무가입).
2. salon_token → 그 살롱 익명 상담 세션(단명 토큰).
3. 언어 → 전화번호(연락처) → 프로필·시술·고민(탭). *타이핑 아닌 탭, 1~2분, 평생 1회.*
4. AI 한국어 요약 → 디자이너 **자체 푸시 브로드캐스트** → 먼저 '받기' 연결+태깅(단골이면 지난 담당 우선).
5. (외국인)번역 상담 → 시술 + 30초 기록(약제·상태·비포애프터) → 리포트 발송(손님 언어).

**재방문 (거의 자동)**
- **같은 폰** → 기기 토큰이 인식 → 번호·인증 없이 "또 오셨네요, 지난번처럼? / 새 스타일?" 탭.
- **새 폰** → 번호 입력 → **OTP 인증** 후 프로필 로드.
- 안 변하는 프로필(얼굴형·모발이력)은 재요청 안 함. 매 방문은 "오늘 뭐?"만, 그것도 지난번이 기본값 → 몇 초.

**데이터 적립**: 전화번호(또는 기기) 키 인바디 이력 누적, salon_id 하위 저장 = 데이터 엔진 연료.

## 9. 알림 & 라우팅

- **메인 알림 = 자체 웹/PWA 푸시 + 딥링크**(상담 직행). 카카오 거치지 않음.
- **iOS 주의**: 웹 푸시는 '홈 화면 추가 PWA'에서만 → 온보딩 현장에서 PWA 설치 + 푸시 권한 + 테스트 1발 필수.
- **라우팅(기본 claim)**: 접수 → 디자이너 폰 브로드캐스트 → 먼저 받기 → 연결·태깅, 나머지 알림 해제. 1명이면 직행. 단골 자동 라우팅. 명시 배정은 옵션. 미수령 시 큐 잔류 + 사장 노출.
- **카카오 알림톡(한정)**: ① 디자이너 초대/기기등록 링크 ② 자체 푸시 백업. *상시 상담 알림 레일 아님.* 자동 알림톡은 비즈채널+템플릿 승인+발송 대행 선행 → 그 전엔 SMS/수동.

## 10. 인증 · 접근 · 보안 *(갱신)*

### 10.1 손님 식별 모델 (수정)
**원칙: 인증 안 된 번호는 절대 이력을 로드하거나 신원에 바인딩하지 않는다.**

| 상황 | 처리 | 인증 |
|---|---|---|
| 같은 기기(토큰 있음) | 자동 인식, 이력 로드 | **없음** (기기 스코프라 그 기기 이력만 노출 → 안전) |
| 새 기기에서 이력 로드 / 번호를 지속 신원으로 바인딩 | 허용 전 본인 확인 | **OTP 필수** |
| 미인증 번호 | 그 방문 연락처 **라벨**로만 사용 | 이력·신원 키 **불가** |

→ X가 Y 번호를 쳐도 OTP 없이는 Y 이력 못 보고, X 기기가 Y 신원에 안 묶임. 진짜 Y가 인증하면 Y 신원은 깨끗(오염·사칭 차단).
→ **기기 토큰은 인증 없이 신뢰 가능**(자기 기기 이력만), **번호는 인증돼야 신원**(아무나 입력 가능하므로). 신뢰 성격이 다름.
→ **외국인 SMS**: 안 터질 수 있어 → 기기 토큰 의존(여행 중 같은 폰), cross-기기는 이메일 OTP 대안.

### 10.2 운영자 3계층
1. **손님** — 위 §10.1.
2. **디자이너·사장** — 신원+역할. 디자이너 = 일회성 기기 등록(사장 추가 → 알림톡/SMS 초대 → 폰 링크 1회). 사장 = `owner`. staff 오픈가입 금지.
3. **플랫폼 어드민** — 내부 전용 + 허용 목록 + 감사 로그.

### 10.3 강제 지점
접근통제는 프론트 아닌 **DB(Supabase RLS)**. 모든 테넌트 행 `salon_id`, 정책이 권한 밖 행을 안 돌려줌. 쓰기 역할별 제한, 감사 로그, URL 민감정보 금지, 토큰 만료·일회성·재발송. v1: 민감 미니 어드민 내부용으로 닫음.

## 11. 기술 아키텍처 & 멀티테넌시

- **프런트**: 단일 Next.js(Vercel). 손님=모바일 웹, 디자이너=PWA+자체 푸시, 사장/어드민=웹. 살롱별 배포 없음.
- **백엔드**: 단일 Supabase — Postgres·RLS·Realtime·Edge Function(Gemini)·Storage.
- **멀티테넌시 = 풀드**: 모든 행 `salon_id` + RLS. **살롱 추가 = 서버 아니라 행 삽입**. 비용 살롱 수와 거의 무관.
- **격리 규율**: 테넌트 테이블 `salon_id` NOT NULL + RLS + 격리 테스트.
- **네이티브(Flutter) 앱**: v1.5+ (리텐션 검증 후 디자이너용부터).

### 11.1 데이터 거버넌스 2층
운영과 학습을 분리한다 — "A살롱 데이터로 B살롱 추천?" 방어.
- **운영 레이어** — `salon_id` RLS 격리. 각 살롱은 자기 데이터만 본다.
- **글로벌 학습 레이어** — 동의·비식별 후 전체 학습용. 모델은 여기서 학습.
- **식별 범위 이원화** — 기본정보(국적·모발타입 등)는 글로벌 학습 / **약제 레시피는 살롱 전용**(영업비밀 보호).

### 11.2 개인정보·동의 설계 (PIPA — day-1 필수)

NOW #4에서 비포/애프터 사진이 필수가 되는 순간, **얼굴(생체 인접) 수집이 출시 즉시 발생**한다. 동의는 체크리스트 한 줄이 아니라 *설계*가 있어야 한다.

**누가 찍느냐는 동의 주체를 안 바꾼다.** 사진은 디자이너가 찍지만 정보주체는 늘 손님 → 질문은 "디자이너가 찍으니 누가 책임지나"가 아니라 **"손님이 찍히기 전에 동의했나"**. 동의 자리는 이미 있다 — **손님 인테이크**(QR로 자기 폰에서 입력하는 그 단계). 거기 동의 한 줄을 넣으면 디자이너가 사진 찍는 시점엔 이미 동의 완료, 디자이너 마찰 0.

**동의 2겹 분리 (핵심)**:
1. **(필수) 상담·기록용** — 이 사진을 오늘 상담과 카르테에 쓴다. 거부 시 진행 불가.
2. **(선택·분리) 학습용** — 비식별로 모델 개선에 쓴다. 별도 체크박스. 거부해도 서비스 동일, 글로벌 학습 레이어(§11.1)에만 안 넣음.

**왜 나누나**: 일반 비포/애프터는 '개인정보'(동의 필요)지만, **얼굴을 모델에 넣어 식별·매칭하면 '생체인식정보'(민감정보 → 별도 명시 동의)**로 넘어간다. 학습용 동의를 **day-1부터 따로** 받아둬야 초기 데이터를 나중에 합법적으로 학습에 쓴다. 안 받으면 day-1 데이터의 학습 가치가 사라짐.

**동의-사진 결합 (A와 한 뿌리)**: 동의·사진·기록이 **같은 손님 세션(기기 토큰, §MVP 신원 모델)에 묶여야** 한다. 따로 올린 사진은 어느 동의에 붙는지 알 수 없음. → 세션 앵커가 재방문 로드(A)와 동의-사진 결합(B)을 동시에 푼다.

**운영 규칙**: **동의 기록 없으면 카메라 단계 비활성화.** 인테이크 건너뛴 워크인은 디자이너 화면에서 손님에게 폰 내밀어 동의받고 진행.

**법적 구조**: 살롱 = 처리자(자기 손님·서비스), 소통 = 수탁자(살롱 대신 저장). 단 **글로벌 학습은 소통의 별도 목적**이라 그 부분만 소통이 직접 동의 → §11.1 2층 거버넌스와 맞물림. *(생체인식정보 해석·의료미용 확장은 개인정보 전문가 확인 필요 — 방향만 제시, 법률 자문 아님.)*

## 12. 데이터 모델 *(갱신)*

전체는 `소통_ERD_명세서.md`. 핵심:

- **인바디형 이력** — `customers` 밑에 방문마다 (날짜·시술·약제·상태 점수·비포애프터) 누적. **신원 앵커 = 기기 토큰 또는 OTP 인증된 전화번호.** 미인증 번호는 연락처 라벨일 뿐, 이력에 안 묶음(merge 금지).
- **소유·격리** — 모든 데이터 `salon_id` 하위. `treatment_record`에 디자이너 필드.
- **사장 다축 조회** — 디자이너별/전체/시간순/회원별 = 필터·정렬 쿼리.
- **다국어** — 카탈로그 `name_ko`(+tr), 동적 `source_text`(+tr).

### 12.1 결과 신호 3종 (모델 학습용 정답지)
만족도 설문에만 의존하지 않는다. 세 가지로 결과를 잡는다.
1. **사진** (비포/애프터) — 객관 정답, 1차.
2. **재방문 행동** — "다시 왔다 = 만족" 행동 신호, 1차.
3. **만족도** — 명시 평가(sparse, 있으면 보조).

→ 이 3종이 §4 (2)(3)층 모델의 **연료이자 정답**. MVP는 최소 사진+재방문이 잡히게 설계.

### 12.2 데이터 수집 동기 — "왜 디자이너가 꾸준히 입력하는가" (행동 설계)

**가장 큰 사업 리스크**: 모든 논리가 "상담 → 데이터 → 모델"인데, 데이터는 디자이너가 입력해야 쌓인다. 디자이너는 바쁘다 — "30초 입력"도 현장에선 길다. 입력이 *회사를 위한 의무*면 안 한다 → 데이터 0 → 모델 0 → 사업 0.

**핵심 원칙**: 입력을 **"의무"가 아니라 "자기 일을 쉽게 하는 행위"로** 만든다. 디자이너가 입력하는 이유는 소통한테 데이터를 주려는 게 아니라 **자기가 편해지려고**여야 한다.

**4가지 장치**:

1. **이미 하는 행동에 얹는다 (추가 부담 0)** — 새 일을 시키지 않는다. 비포 사진·애프터 사진을 소통 안에서 찍으면 그게 곧 데이터. 약제·시술은 차트에 적던 것 → 탭 몇 번. *(단, "모든 살롱이 비포/애프터를 원래 찍는다"는 가정이 아니라 — 촬영 습관 유무 자체를 파일럿에서 검증한다. §15.1.)*

2. **입력의 보상을 재방문 때 돌려준다** — 입력을 "사후 기록"이 아니라 "다음 손님 준비"로 프레임. 재방문 손님이 오면 **지난 기록이 자동으로 뜬다** → 디자이너가 "지난번에 입력해둬서 지금 편하다"를 체감 → 다음 입력 동기. *미래의 나를 돕는 행위.*

3. **핵심 데이터는 입력 없이 자동 수집** — 결과 신호 3종(§12.1) 중 **재방문은 행동이라 저절로 잡힌다**(손님이 또 오면 시스템이 앎, 디자이너 입력 0). 만족도 설문에 의존하지 않음으로써 입력 동기 문제를 *자동 수집으로 우회*.

4. **재방문 시 상담 시간 단축 (진짜 큰 보상)** — 재방문 손님이 오면 입력해둔 기록으로 **지난 약제·염색 배합·주의사항·알레르기가 자동 표시** → 처음부터 묻고 파악하던 상담이 **절반으로 단축**. "리포트 만들어줌"보다 *"내 상담 시간이 반으로 준다"*가 디자이너한테 훨씬 직접적인 이득. (부차 보상: 입력 직후 손님 언어 리포트 자동 생성 → 손님 만족·리뷰·재방문. *간소형 리포트는 동기 장치라 MVP 포함, 고도화는 NEXT.*)

**VC 질문 "데이터를 누가·왜·꾸준히 입력하나?"의 10초 답**:
> "디자이너는 데이터를 입력하는 게 아니라, **자기 손님을 더 잘 관리하려고** 소통을 쓴다. 데이터는 그 부산물로 쌓인다. 입력 안 하면 다음 방문 때 자기가 손해다."

**→ MVP 설계 지시 (이 동기를 구현으로)**:
- 결과 입력은 **사진 중심**, 텍스트 최소
- 재방문 **자동 감지** (입력 의존 X)
- 재방문 시 **지난 약제·배합·주의 자동 표시** → 상담 시간 단축 (핵심 보상)
- 지난 기록을 **기본값으로 띄우기** (입력 마찰 최소 — 같으면 탭 1번)

**압축 한 문장 (전체 동기를 한 줄로)**:
> **우리는 데이터를 수집하려는 회사가 아니다. 디자이너의 업무를 줄이다 보면 데이터는 자연스럽게 남는다.**

**입력 제거 로드맵 (궁극 방향)**:
- *입력을 "잘 하게" 만드는 것보다 "안 하게" 만드는 게 더 좋은 제품.* 궁극적으로 AI가 상담 내용을 듣고 자동 기록 → 디자이너는 **확인 버튼만**. 입력을 *없애는 것*이 최종 목표. (단계: 지금=탭 입력 최소화 → 다음=음성/대화 자동 기록 → 최종=확인만.)

## 15. 측정 & 지표

파일럿에서 '내 매출'은 측정 대상 아님. 보조 3종: ① 유지·전환율 ② 살롱당 완결 레코드 ③ 살롱 가치배수(증분 매출 ÷ 구독료). *단 최상위 지표는 North Star=상담 성공률(★MVP 섹션), 이 3종은 그 하위 보조지표.*

### 15.1 핵심 가설 & 검증 계획 (숫자는 파일럿이 채운다)

현재 PRD의 핵심 주장은 대부분 *검증 전 가설*이다. 숫자를 지어내지 않고, **무엇을·어떻게 잴지**를 명시한다. (정직 규율: 가정을 단정으로 쓰지 않는다.)

| # | 핵심 가설 | 지금 상태 | 검증 방법 | 합격선(예시) |
|---|---|---|---|---|
| H1 | 외국인 워크인이 충분히 많다 | 매크로만 입증(18.5%·5배), 개별 살롱 농도 미검증 | 파일럿 살롱에서 **일 외국인 수 실측** | 일 평균 외국인 ≥ N명 |
| H2 | 외국인 한두 명이면 구독료 회수 | 객단가 가정(10만+) | 파일럿에서 **놓친 외국인 → 전환 손님 수** 측정 | 월 전환 ≥ 구독료/객단가 |
| H3 | 입력 부담이 작다("30초") | **측정 안 됨** — 희망사항 | 실제 디자이너 **5명에게 입력 시간 실측** + 인터뷰 | 평균 입력 ≤ X초, 이탈률 낮음 |
| H4 | 비포/애프터 촬영 습관이 있다 | 일부 살롱 가정 | 파일럿 살롱 **촬영 습관 유무 관찰** | 촬영률 ≥ Y% |
| H5 | 재방문 기록이 입력 동기가 된다 | 논리 가설 | 재방문 시 **상담시간 단축 실측** + 디자이너 체감 인터뷰 | 상담시간 ↓ + 재입력 지속 |

**검증 순서**: H3·H4(입력이 실제로 일어나나)가 **1순위** — 이게 안 되면 H1·H2·H5가 다 무의미. 첫 파일럿(작은아빠네)에서 H3·H4부터 깬다.

**VC 대비**: "외국인 하루 몇 명? 몇 명 놓침? 객단가? 입력 몇 초?" → 지금은 "파일럿에서 이렇게 잰다"가 답. 파일럿 후엔 실측값으로 교체.

## 16. GTM / 파일럿

- 비치헤드: 신사·청담·홍대·성수·명동. 파일럿 3곳 무료 파일럿(기간 미확정) → 유료 전환. **피칭은 외국인 웨지로**(가설 깨끗하게), 기술적으론 한국인 기록도 열어둠(데이터·끈끈함 보너스).
- **온보딩(살롱)**: 살롱 행 + QR + 사장 계정 → 사장이 디자이너 추가 → 자동 알림톡/SMS 초대 → 기기 등록 + 현장 PWA 설치·푸시·테스트.
- 영업 자산(제작 중): 사장님 소개 1장, 공모전 3단 리플릿, 디자이너/사장 퀵스타트.

### 16.1 Pilot Exit Criteria (파일럿 종료·전환 판단 조건)
파일럿을 "성공"으로 보고 다음 단계(유료 확장)로 갈 조건:

| # | 조건 | 기준(목표) |
|---|---|---|
| ① | 살롱 수 | 3곳 운영 |
| ② | 누적 상담 | 100건 |
| ③ | **입력 유지율** | 70% 이상 — *지속률*(시간이 지나도 디자이너가 계속 입력하는가). cf. 완결률(첫달 Success)=레코드 완전성(상담→결과+사진까지 채워진 비율). 둘은 다른 지표. |
| ④ | 유료 전환 | 2곳 이상 |

→ ③ 입력 유지율이 가장 중요한 게이트. 이게 안 나오면 기능을 늘리지 말고 **입력 동기 설계(§12.2)를 다시 판다.** 숫자는 목표 가설, 실측으로 보정.

### 16.2 Launch Checklist (출시 전 운영 점검)
```
[ ] 살롱 QR 제작·비치 (입구·대기·테이블텐트, 다국어 안내카드)

[ ] 다국어 입력 테스트 (영/일/중)

[ ] 개인정보 동의 화면 (PIPA) — **필수(상담·기록) + 선택(학습) 2겹**, 인테이크에 배치 (§11.2)
```

### 최종 목표 (엔드 비전)
소통이 쌓는 '실제 한국 살롱 시술 결과' 데이터로 **K-뷰티 시술 결과 예측 모델**(§4-③)을 학습 → 얼굴·모발만 보고 시술/스타일과 그 결과·만족도를 예측·추천하는 **표준 엔진**이 된다. 이 모델이 큰 그림의 심장.

### 큰 그림 — 제품군 엔드게임
- **소통** (전 고객 접수·카르테·전환·예약) = 데이터 엔진
- **MAKEDOL** (열망·변신 콘텐츠) = pull, 모델의 변신 쇼케이스
- **수요 기계**: MAKEDOL(열망) → 소통(발견·전환·예약)
- 수요를 쥐면 거래 수수료(예약금·take-rate)로 수익 전환 → 카테고리(의료미용)·해외(일본)로 확장

**한 줄**: 소통은 *도구로 들어가* 데이터를 쌓고 → *모델로 깊어지고* → *수요를 쥐며 플랫폼이 된다.*

### 비전 여정 (수렴 구조) — `소통_큰그림_수렴.html`
Sotong을 허브로 두 갈래가 하나로 모인다.

- **수요·문화 만들기**: 헤어부터(소통) → 외국인이 한국 헤어 시술을 쉽게 받는 환경 → MAKEDOL → 외국인의 한국 뷰티시술이 *방한 필수 코스로 일상화* → 소통으로 모두(외국인+한국인)를 기록·관리.
- **도메인 확장**: 소통이 헤어를 넘어 뷰티샵(메이크업·네일)·피부과·성형외과로.
- **수렴 → 모델**: 두 갈래가 **'한국 시술 데이터(외국인 + 한국인 · 전 도메인)'**로 합쳐짐 → AI 학습 → 개인화 얼굴 기반 모델. 모델은 **두 출력**을 낸다:
  - **출력 ① 판단 (예측)**: "이 얼굴·모발 + 이 시술 → 실제로 이런 결과·만족도" (의사결정 보조)
  - **출력 ② 생성 (시각화)**: 판단이 정한 결과를 **기존 얼굴에 편집·생성**해 시술 전 미리보기 (체험·전환). MAKEDOL의 변신 쇼케이스가 곧 이것.
  - 둘 다 **실제 한국 살롱 결과 데이터 기반** — 일반 합성 앱과 선을 긋는 해자. 생성(②)은 판단(①)에 종속(§4.1 파이프라인). **판단 먼저 → 생성 나중** 순서.

### 모델 적용 범위 (왜 큰가)
**지배 규칙: 모델은 그 뒤의 데이터만큼만 옮겨간다.** "어디 적용되냐" = "어디서 결과 데이터를 얻고 수요가 있냐".

- **즉시 (제품군 안)** — 소통(살롱 내 추천·대기시간 분석·손님↔살롱 매칭) · MAKEDOL(얼굴→한국식 변신). 이미 집이 둘.
- **같은 패턴 확장 (새 데이터 필요)** — 메이크업 → 피부·성형(의료미용). 아키텍처는 전이되나 *데이터는 도메인마다 새로* + 의료는 규제. 복붙 아님.
- **모델 라이선싱 (나중, 큰 판)** — 뷰티 브랜드·커머스·미용 교육이 '얼굴 기반 개인화'를 원함. 결과 근거 버전은 우리 데이터 없이는 불가 → 데이터→모델→라이선스.
- **지리적** — "한국식 스타일 결과"는 글로벌 수요(K-뷰티·한류). 모델은 국경 독립 → 일본·중국·동남아.

종착: 소통은 살롱 도구가 아니라 **K-뷰티 개인화 레이어(인프라)**가 된다.

> **규율**: 이 넓이는 *'왜 큰가'의 서사*지 지금 할 일이 아니다. 실행은 살롱 헤어 하나 깊게(데이터 엔진부터). 넓이는 피칭 업사이드, 손은 깊이 먼저.

## 21. 미결정 사항 *(갱신)*

**오늘 닫힘**: 타깃(외국인 웨지 + 전 고객 확장) · 손님 식별 모델(기기 토큰/인증 번호/미인증 라벨) · 재방문 UX · 외국인 SMS 대응.

**여전히 열림**:
1. 살롱당 외국인 워크인 **농도** — 파일럿이 답.
2. **정가 및 무료 파일럿 기간** (미확정 — 현장 검증으로 확정) + 전환률.
3. 명시 배정 vs claim 기본값.
4. 회원(손님) 식별 범위 — **살롱별 vs 글로벌**(번호/이력 공유 여부).
5. 중국어 노출 시점.
6. 이름/도메인·상표·핸들.
7. 중앙대 창업지원 **자격**(예비 vs 기창업).
